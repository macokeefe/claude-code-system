## Workflow

For every task, follow this sequence. Do not skip steps.

### 1. Read First

- What you're going to build or change
- Which files will be created or modified
- Any tradeoffs or risks

- One concern per commit
- Don't refactor while adding a feature (separate PRs)

### 4. Verify

- If tests don't exist, write them
- Check for TypeScript errors before calling a task done

### 5. Report Back
When done, summarize:
- What was built
- What files were changed
- Any known limitations or follow-up tasks

- **Push to production** — no `git push origin main` or deployment commands
- **Modify authentication logic** — auth changes require human review

- **Install new dependencies** — all new packages require team approval
- **Modify billing or payment code** — any Stripe or payment logic is off-limits without explicit sign-off
- **Send emails or notifications** — do not trigger real sends in development; use test mode
- **Access external APIs with real credentials** — use sandbox/test keys unless told otherwise
- **Refactor while building features** — keep concerns separate across PRs
- **Merge or resolve conflicts autonomously** — always surface conflicts for human resolution

## Architecture

**Request flow:**
1. Browser → Next.js middleware (Clerk auth check)
2. Server component fetches data via tRPC server caller (no HTTP round-trip)
3. Client components call tRPC procedures via React Query hooks
4. tRPC procedures call Prisma → Postgres

**Multi-tenancy model:**
- Every resource belongs to an `Organization` (Clerk org)
- Every DB query MUST filter by `organizationId`
- Always derive `organizationId` from Clerk session, never from request body

**Stripe integration:**
- The webhook handler at `/api/webhooks/stripe` is the ONLY place that updates subscription status
- Feature gates check `organization.subscriptionStatus === 'active'`

## API Conventions

**tRPC conventions:**
- One router file per domain: `users.ts`, `invoices.ts`, `billing.ts`
- All procedures that read or write org data: `const { organizationId } = await requireOrg(ctx)`
- Use `protectedProcedure` for anything requiring auth
- Zod schemas for all inputs

**Auth middleware pattern:**
```typescript
const { userId, organizationId } = await requireOrg(ctx);
const item = await db.item.findUniqueOrThrow({ where: { id: input.id, organizationId } });
```

---

## Database Conventions

- Every schema change requires a Prisma migration
- Tables: plural snake_case. Columns: camelCase in Prisma (maps to snake_case via `@map`)
- Soft deletes: use `deletedAt DateTime?` — do NOT use hard deletes for user data
- Always index foreign keys and columns used in WHERE clauses

## Authentication

### API Key Auth

- Key format: `sk_live_...` (production), `sk_test_...` (test)

## Key Endpoints

### `GET /v2/users/{id}`
**Purpose:** Fetch a single user record by vendor ID.
**Auth:** Bearer token in header.
**Known quirks:** `created_at` is UTC but lacks timezone suffix in some older records.

### `POST /v2/charges`
**Purpose:** Create a payment charge.
**Idempotency:** Send `Idempotency-Key` header (UUID v4 per logical operation, not per retry).
**Async:** Status is `pending` immediately; final state delivered via webhook.

---

## Rate Limits

| Endpoint / Scope | Limit | Window | Notes |
|-----------------|-------|--------|-------|
| Global | 1,000 req | per minute | Shared across all API keys in org |
| `POST /charges` | 100 req | per minute | Per customer |
| `GET /reports/*` | 10 req | per minute | Cache aggressively |

### Handling 429s
- The API returns `Retry-After: ` on 429 — always respect it.
- Retry strategy: exponential backoff with jitter.
  - Base delay: 1s, max delay: 32s, max attempts: 5.
  - Jitter: `delay = base * 2^attempt + random(0, 1000ms)`.
- Never retry non-idempotent requests without a stable `Idempotency-Key`.

## Error Handling Conventions

| HTTP Status | Meaning | Action |
|-------------|---------|--------|
| 400 | Bad request (our bug) | Log full request, alert dev team, do NOT retry |
| 401 | Auth failure | Check/rotate credentials, alert ops |
| 403 | Permission denied | Check OAuth scopes, do NOT retry |
| 404 | Resource not found | Return `null` to caller, not an exception |
| 409 | Conflict / duplicate | Check idempotency key logic |
| 422 | Validation error | Surface `error.message` to the user |
| 429 | Rate limited | Retry with backoff |
| 5xx | Vendor outage | Retry with backoff; open circuit after 5 consecutive failures |

**Application-Level Conventions:**
- Catch API errors at the integration layer.
- Transform to internal error types: `ApiAuthError`, `ApiRateLimitError`, `ApiValidationError`, `ApiServerError`.
- Log with structured fields: `{ vendor, endpoint, status, error_code, request_id, duration_ms }`.
- Never log request bodies that may contain PII or payment data.

---

## Data Transformation

Raw API responses are transformed into internal domain models at a single boundary. Do not scatter transformation logic across the codebase.

**Rules:**
- All date strings are parsed and stored as UTC datetimes.
- Monetary amounts are in **cents (integer)** — format at the presentation layer.
- Unknown/null fields are mapped to `null` in the internal model, not omitted.

## Caching Strategy

| Data | TTL | Invalidation Trigger |
|------|-----|---------------------|
| User profile | 5 min | Webhook `user.updated` event |
| Product catalog | 1 hour | Manual flush or deploy |
| Exchange rates | 10 min | Time-based only |
| Auth tokens | `expires_in - 60s` | Token refresh |

**What NOT to cache:**
- Payment/transaction status (always fetch fresh)
- Any response with `Cache-Control: no-store`

**Implementation:**
- Cache key pattern: `{service}:{resource}:{id}` e.g., `stripe:customer:cus_abc`
- On cache error: log and fall through to API — never fail because the cache is down.

## Testing Approach

**Rule: No test should ever hit a live production API.**

1. **Vendor sandbox / test environment** — use for integration tests.
2. **Recorded fixtures** — record real API responses once, replay in tests.
3. **Mocks** — mock the HTTP client for unit tests.

**Fixture Management:**
- Fixtures live in `tests/fixtures/api/{vendor}/{endpoint}/`.
- File naming: `{scenario}.json` e.g., `charge_success.json`.
- Fixtures must never contain real API keys, PII, or production IDs.

## Security Rules

1. Never log API keys or secrets.
2. Never expose vendor API keys to the frontend.
3. Webhook signature verification is mandatory — always verify before processing.
4. Treat webhook payloads as untrusted — validate all fields.
5. Rotate keys on any suspected exposure immediately.
6. Use least-privilege OAuth scopes.
7. Rate limit inbound requests to prevent abuse of endpoints that trigger vendor calls.
8. Store secrets in a secrets manager — never in committed files.
9. Audit log all mutating operations.

## Common Patterns

### Pagination
```
GET /v2/charges?limit=100&starting_after=ch_lastId
```
- Loop until `has_more: false`, processing page-by-page.
- Never assume a stable sort order across pages.

### Webhooks
- Endpoint must return `200` within 5 seconds — enqueue heavy processing.
- Deduplicate by event ID.
- Replay failed webhooks: keep a log of processed event IDs.

### Circuit Breaker
- After 5 consecutive failures, open the circuit.
- After 30s in open state, allow one probe request.
- If probe succeeds, close the circuit. If it fails, re-open.

## Runbook

### Debugging a failed API call
1. Find the `request_id` in logs.
2. Check vendor status page.
3. Reproduce in sandbox.
4. Check if the issue is auth (401), bad payload (400/422), or vendor-side (5xx).

### Adding a new endpoint
1. Add the raw HTTP call in the adapter layer.
2. Add the response type and transformation function.
3. Record a fixture.
4. Write tests covering success, 4xx, and 5xx cases.
5. Add new env vars to this file and deployment config.

### Rotating an API key
1. Generate new key in vendor dashboard.
2. Update in Secrets Manager / Vault.
3. Trigger secret refresh or redeploy.
4. Monitor error rate for 5 minutes.
5. Revoke old key.
