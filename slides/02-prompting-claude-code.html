Claude Code — Prompting

Prompting Claude Code

5 prompts. Every session covered.

The Problem

ChatGPT habits don't work here

ChatGPT Style

Ask once. Expect an answer.

"Add a login form to my app"
Claude generates something generic
You spend 20 minutes adapting it
Half the session is correction
Claude Code Style

Set context. Ask Claude to plan. Then build.

Brief Claude on your full project first
Ask it to read files before writing
Request a plan before implementation
First output fits — minimal correction needed

Claude Code is a long-context coding agent, not a chatbot. Treat the session like a briefing, not a Q&A.

PROMPT 1 / 5 WHEN TO USE: Start of every session

Project Kickoff

# Project Kickoff Prompt I'm working on [project name] — [one sentence description]. Stack: TypeScript, React, Vite, Tailwind Key files: src/App.tsx, src/components/, src/store/ Before we start, please: 1. Read src/App.tsx to understand the current structure 2. Read src/store/ to understand state management 3. Summarize what you understand about the project 4. Ask any clarifying questions before we build anything Today's goal: [what you want to accomplish this session]

Always ask Claude to summarize back before building — it surfaces misunderstandings before they become wrong code.

PROMPT 2 / 5 WHEN TO USE: Something broke

Debug Mode

# Debug Mode Prompt I have a bug. Here's exactly what's happening: Expected: [what should happen] Actual: [what is happening instead] Error: [paste the full error message] Context: - This started happening after: [what changed] - Relevant file: [file path] Before suggesting a fix, please: 1. Read the file and understand its current state 2. Explain what you think is causing the bug 3. Ask if your theory matches my understanding 4. Then propose the fix

Making Claude explain its theory first catches wrong diagnoses before they produce wrong fixes.

PROMPT 3 / 5 WHEN TO USE: Before every commit

Code Review

# Code Review Prompt Please review [file or diff] before I commit this. I want you to check specifically for: 1. Bugs or logic errors 2. Edge cases I haven't handled 3. Anything that will break with unexpected input 4. Security issues (if relevant) What this code does: [one sentence] Context: [anything reviewers need to know] Please be direct. Don't soften feedback. Flag anything you're uncertain about — don't just skip it because you're not sure.

"Don't soften feedback" matters — Claude defaults to diplomatic. You want honest, not kind.

PROMPT 4 / 5 WHEN TO USE: Cleaning up existing code

Refactor

# Refactor Prompt I want to refactor [file or component name]. Goal: [what you want the refactored version to achieve] Constraints (do not change these): - The external API / props interface must stay the same - Do not change [specific thing] Before making any changes: 1. Read the current file 2. Describe what you plan to change and why 3. Call out any risks or side effects 4. Wait for my confirmation before writing code

The plan-first step catches scope creep before it happens — Claude's refactors tend to expand unless you constrain them.

PROMPT 5 / 5 WHEN TO USE: Building something new

Feature from Spec

# Feature from Spec Prompt I need to build: [feature name] What it should do: - [behavior 1] - [behavior 2] - [behavior 3] What it should NOT do: - [anti-requirement 1] Files that will likely be involved: [list the files] Before writing any code: 1. Read those files to understand existing patterns 2. Propose an implementation plan 3. Flag any questions or unclear requirements 4. Wait for approval before proceeding

Anti-requirements ("what it should NOT do") are underrated. They prevent half the wrong implementations.

Sequencing

Prompts stack. Use them in order.

1

Kickoff — Set the full context

Run this at the start of every session. Claude reads your files and understands the project before touching anything.

2

Feature from Spec — Plan before building

Use this when you know what you want to build. Forces Claude to plan and surface questions before generating code.

3

Code Review — Before every commit

Run a review on the diff before you commit. Catches bugs and edge cases when they're still cheap to fix.

4

Debug Mode — When something breaks

Use this instead of pasting errors and hoping. Structure forces Claude to diagnose before it prescribes.

Warning Signs

When something feels off, there's a pattern

Warning

Claude keeps changing things you didn't ask it to change

Context drift. Claude's mental model of your project has slipped. Reset with the Kickoff prompt — make it read the current files again.

Warning

Answers getting vaguer and more hedged with each exchange

Context window is filling up. Start a fresh session. Don't try to push through — quality degrades fast near the limit.

Warning

Claude's suggestions contradict code it wrote earlier in the session

Session drift — Claude has lost track of its own earlier work. New session, Kickoff prompt, point it at the current state of the files.

Anatomy

Every good prompt has the same five parts

CONTEXT
What project, what files, what's already built
GOAL
Exactly what you want to accomplish — specific, not vague
CONSTRAINTS
What must not change. Anti-requirements. Hard limits.
PROCESS
Read first → plan → confirm → then build
TONE
Be direct. Don't soften. Flag uncertainty.
# Applied example # CONTEXT Working on the GTO Poker app. Read src/analyzer/hand.ts first. # GOAL Add equity calculation to HandAnalyzer # CONSTRAINTS Don't change the HandAnalyzer interface. Don't add new dependencies. # PROCESS 1. Read hand.ts 2. Propose the plan 3. Wait for approval # TONE Be direct. Flag any uncertainty.
The Pattern

Every prompt delays coding until Claude understands the full picture.

That delay is the feature, not the friction. The ten seconds you spend writing a structured prompt saves thirty minutes of correcting output that was confidently wrong from the start.

Claude doesn't know what you know. These prompts are the mechanism for closing that gap before the first line of code gets written.
