Debugging with Claude Code

Stop guessing. Start closing the loop.

The failure pattern

The loop that kills your momentum

Claude guesses

Generates a fix based on incomplete context

↓

You apply it

Copy-paste, run, test

↓

Still broken

Different error, or same error, or new error

↓

Claude guesses again

No new information. Another shot in the dark.

↻ repeat
The core principle

Hypothesis before solution.

Before Claude writes a single line of fix code, it should state what it thinks is wrong and why. A named hypothesis is testable. A guess is not.

This one shift turns debugging from a slot machine into a process.

THE PROMPT
Debug mode template
Enter debug mode. Do NOT write any code yet. First, state your hypothesis: - What do you think is causing this bug? - Why do you think that? - What evidence supports it? Then propose ONE test to validate your hypothesis before we fix anything. --- BUG REPORT Expected: [EXPECTED BEHAVIOR] Actual: [ACTUAL BEHAVIOR] Error: [PASTE ERROR MESSAGE / STACK TRACE HERE] What I've tried: [WHAT YOU ALREADY ATTEMPTED] Context: [RELEVANT FILE / FUNCTION / COMPONENT]
Step 1

Define the problem precisely

01 — Expected

Expected behavior

What should happen? Be specific. "The button submits the form and redirects to /dashboard" beats "it should work."

02 — Actual

Actual behavior

What actually happens? Include the exact error message, not a paraphrase. Verbatim is better.

03 — Error

Error message

Full stack trace if available. Include the first line AND the line that points to your code — both matter.

04 — Tried

What you've tried

Eliminates dead ends. Stops Claude from suggesting the exact thing you already tried five minutes ago.

Step 2

Form a hypothesis

Good hypothesis

Specific, testable, points to a mechanism. You can design one experiment to validate or refute it.

"The JWT token is expiring before the API call completes because the client clock is off by 5 minutes. Test: check token exp claim vs server time."

Bad hypothesis

Vague, untestable, or just restates the symptom. There's no experiment you could run to confirm it.

"Something is wrong with the auth. Maybe the token isn't working right or there's a network issue somewhere."
Step 3
One thing.

Every time you change multiple things at once, you break the feedback loop. If it works, you don't know why. If it doesn't, you don't know why either.

Change one variable per test. Confirm or refute the hypothesis. Then move to the next candidate.

Step 4

Close the loop

It works →

✓

Commit with a message explaining what was wrong, not just what changed

✓

Document the root cause in a comment near the fix

✓

Update CLAUDE.md if this reveals something about your stack's behavior

It doesn't →

→

Refine the hypothesis with what the test result told you

→

Move to the next candidate in the hypothesis list

→

If three tests fail, consider resetting Claude's context entirely

Context rot

When to reset and start fresh

Signal: circular suggestions

Claude keeps recommending things it already suggested, or contradicting its own earlier advice.

Reset: start a new conversation, paste only the minimal bug report

Signal: hedging everything

"It could be X, or maybe Y, or possibly Z" with no confidence ranking. Claude is guessing randomly.

Reset: fresh context, force hypothesis format from the start

Signal: 5+ failed fixes

You've been in the same conversation for too long. Context is noisy and Claude is anchored to bad assumptions.

Reset: new session, bring only the relevant files + error
Regression bugs

The hardest kind

Regressions are bugs that didn't exist before. Something you changed broke something that worked. Claude needs to know the before-state to reason about the after-state.

Always tell Claude what changed recently — a dependency bump, a refactor, a merge. The diff is the hypothesis seed.

This is a regression bug. It worked before, now it doesn't. What changed recently: - Upgraded react-navigation 6 → 7 - Merged the auth refactor PR - Updated Node 18 → 22 Last known good state: commit abc123 (2 days ago) Error now: TypeError: Cannot read property 'navigate' of undefined at HomeScreen.tsx:47 Hypothesis before fix, please. Which change most likely caused this?
Performance debugging

Measure before you fix

Performance bugs need a different approach. You can't fix what you haven't measured. Claude needs numbers, not vibes — "feels slow" is not a bug report.

Profile first, then bring Claude the data. A flame graph screenshot plus the specific bottleneck is 10x more useful than a description.

Performance issue — measure first mode. Do NOT suggest fixes yet. Symptom: FlatList scrolling drops to ~30fps with 200+ items Measured: - JS thread: 12ms avg frame time - Re-renders: 47 per scroll event - Memory: 280MB (up from 90MB) Profile output: [paste Flipper / Instruments output] What's causing the re-render spike? Where should I look first?
Build & config errors

What to paste. What not to.

Paste this

✓

The exact error line from the terminal — not a screenshot

✓

The 5–10 lines of stack trace pointing to your code

✓

The relevant config section (package.json, tsconfig, build config)

✓

The command you ran to trigger the error

Don't paste this

✗

The entire 400-line build log — Claude will anchor on the wrong part

✗

Unrelated config files that aren't part of the error path

✗

Files you didn't recently change — noise, not signal

✗

A vague description instead of the actual error text

The debug log

Write it down. Every time.

A debug log is a running record of what you tried, what happened, and what you concluded. It's the single best tool for not repeating yourself — in this session and the next.

Paste it into Claude at the start of a new session and you instantly have full context without reliving the whole conversation.

## Debug Log — 2026-03-24 ## Bug: JWT auth failing intermittently 10:15 — Started investigation Error: 401 Unauthorized on /api/lessons Happens ~30% of requests, not all 10:32 — Hypothesis 1 Token expiry race condition? Test: log token exp before each request Result: tokens valid, not expired → Refuted 10:51 — Hypothesis 2 Clock skew between client and server? Test: compare Date.now() vs server time header Result: 4.3 second delta found → Confirmed! Fix: use server time for exp check 11:04 — Fixed. Committed abc789.
