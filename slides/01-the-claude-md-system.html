The CLAUDE.md System

How to give Claude Code a persistent memory

LEVEL 2 — BUILDER
What it is

A plain text file Claude reads at the start of every session

You put it in the root of your project. Claude picks it up automatically. It tells Claude who you are, what you're building, and how you work — so you don't have to repeat yourself every session.

No plugins. No config. Just a markdown file named CLAUDE.md.

# CLAUDE.md ## Project eMotus — science learning app ## Stack Expo 54, React Native, TypeScript ## Commands npx expo start npm run build ## Current Focus Building the lesson card component
Why it matters

Same Claude. Very different sessions.

Cold Session

Claude knows nothing about your project.

Asks what framework you're using
Guesses at your file structure
Suggests wrong commands
Ignores your naming conventions
With CLAUDE.md

Claude walks in already briefed.

Knows your stack and dependencies
Follows your naming conventions
Runs the right dev commands
Knows exactly what you're building now

The file does nothing fancy — it just removes the warm-up tax from every session.

Contents

Six sections. That's all you need.

Project

What you're building

One sentence. Name, purpose, stage.

Stack

Language, framework, key deps

Only what Claude needs to know to write correct code.

Commands

Dev, build, test scripts

The exact commands Claude should run — no guessing.

Conventions

Naming, structure, patterns

How files are named, how components are structured, what patterns you follow.

Don'ts

What Claude should never do

Hard rules. Never change this file. Always ask before deleting.

Current Focus

What you're building right now

One sentence that anchors every session to your actual goal today.

Scope

Keep it lean. It's a briefing, not a wiki.

Put In
Project name and one-line description
Tech stack and key dependencies
Dev, build, and test commands
File naming and code conventions
Hard constraints and don'ts
Current active task (1 sentence)
Leave Out
Full feature specs or PRDs
Detailed business logic
API docs and schema definitions
Meeting notes or decisions log
Historical context / "how we got here"
Anything longer than 2–3 lines per section
Template

Universal Template

COPY-PASTE READY
# CLAUDE.md ## Project [App name] — [one sentence: what it does and who it's for] ## Stack - Language: [TypeScript / Python / etc] - Framework: [React / Next.js / FastAPI / etc] - Key deps: [list only the important ones] ## Commands - Dev: npm run dev - Build: npm run build - Test: npm test ## Conventions - Components: PascalCase, one file per component - Styles: [CSS modules / Tailwind / styled-components] - [Any other important patterns] ## Don'ts - Never modify [critical file] - Never use [pattern you hate] - Always ask before deleting files ## Current Focus [One sentence: exactly what you're building in this session]
Variant

React Native / Expo

The universal template works. You just need a few RN-specific additions:

Specify Expo version — it matters for API compatibility
Name the navigation library (React Navigation vs Expo Router)
Note iOS vs Android differences where relevant
Add tunnel/device commands if you use them
# RN / Expo additions ## Stack - Expo SDK 54 - React Native 0.76 - Navigation: Expo Router v4 - State: Zustand ## Commands - Dev: npx expo start - iOS sim: npx expo run:ios - Tunnel: npx expo start --tunnel ## Conventions - Routes in app/ folder (file-based) - Shared components in components/ - iOS-first, Android parity second
Variant

React / Vite / TypeScript

Web apps have a simpler setup but benefit from being explicit about CSS approach, routing, and state management.

Specify CSS approach — Tailwind vs modules vs styled-components
Name the router if you use one (React Router, TanStack Router)
List env var names if Claude might need them
Clarify TypeScript strictness level
# React + Vite additions ## Stack - React 18 + TypeScript (strict) - Vite 5 for bundling - Tailwind CSS 3 - React Router v6 - Zustand for state ## Commands - Dev: npm run dev (port 5173) - Build: npm run build - Preview: npm run preview ## Conventions - Tailwind only — no custom CSS - Named exports, no default exports - Strict TS — no 'any'
Mistakes

Four ways to write a CLAUDE.md that doesn't work

1

Too vague — "Build a web app with React"

Claude gets no signal about your actual project. It fills in the blanks with generic assumptions and ignores the file entirely within a few exchanges.

2

Too long — 300+ lines of documentation

Long CLAUDE.md files burn context window space before you've typed a single prompt. Aim for under 60 lines total.

3

Stale content — never updated after week one

Claude works off the wrong stack, wrong commands, wrong focus. An outdated file is worse than no file because it actively misleads.

4

Missing commands — no dev/build/test scripts listed

Claude guesses. It might run yarn when you use npm, or miss your custom scripts entirely.

Maintenance

Update it when the project changes

Trigger

New dependency added

Update the Stack section. Include the package name and version if it matters for API compatibility.

Trigger

Convention changed

Update Conventions immediately. Claude will keep using the old pattern if you don't — and it'll look consistent with your history, making it hard to catch.

Trigger

Starting a new feature

Update Current Focus before you start the session. This one field alone anchors everything Claude does in that session.

Treat it like a README — a living doc, not a one-time setup

Advanced

Per-folder CLAUDE.md for monorepos

In a monorepo, a single root CLAUDE.md doesn't give Claude enough context to work confidently in each package. Add a CLAUDE.md inside each major folder.

Claude reads both — the root for global context, the local one for the specific package. Inner files override outer ones when there's a conflict.

Works great for: frontend + backend split, multiple apps sharing a design system, packages with different test setups.

# Monorepo structure project-root/ ├── CLAUDE.md # global ├── apps/ │ ├── web/ │ │ ├── CLAUDE.md # web app │ │ └── src/ │ └── mobile/ │ ├── CLAUDE.md # RN app │ └── src/ └── packages/ └── ui/ ├── CLAUDE.md # design sys └── src/ # Each inner file only covers # what's different from root
The One Field That Matters Most

Current Focus changes the entire session

Every other section is static background. Current Focus is dynamic. It tells Claude what to optimize for, what to keep asking about, and what to ignore. Without it, Claude treats all context equally — which means it treats nothing as urgent.

One sentence. Updated before each session. That's it. It's the difference between Claude helping you build and Claude generating plausible-looking tangents.
