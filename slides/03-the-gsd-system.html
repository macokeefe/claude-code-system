What GSD Is

A structured execution framework for Claude Code

GSD (Get Stuff Done) is a set of slash commands that impose structure on your build sessions. Instead of freewheeling prompts, you work in defined phases with tracked tasks.

Every session starts with a plan. Every plan has phases. Every phase has a clear deliverable. You always know exactly where you are and what's next.

Phase

A named milestone with a defined scope

↓

Tasks

Concrete, testable work items inside the phase

↓

Execute

Claude runs tasks one at a time, state is saved

Installation

Get it in 30 seconds

GSD ships as part of the superpowers plugin for Claude Code. One command drops all the slash commands into your global config.

Once installed, the commands are available in every project — no per-project setup required.

Your state files live in the project directory, so they're portable and version-controllable.

# Install the superpowers plugin claude mcp add superpowers # Or manually via npm npm install -g @superpowers/claude-code # Verify it's working claude /gsd:status # What gets added: /gsd:new-project → initialize a project plan /gsd:resume-work → pick up where you left off /gsd:execute-phase → run the current phase /gsd:progress → see what's done /gsd:quick → lightweight single-task mode
Command

/gsd:new-project

1

What GSD asks you

Project name, description, tech stack, and success criteria. It builds context before generating anything.

2

What it generates

A phased project plan with 3–6 phases, each containing 4–8 specific, testable tasks. Saved to gsd-state.json.

3

How phases are structured

Each phase has a name, a goal, and a list of tasks. Tasks have a status (todo / in-progress / done) and an optional acceptance test.

4

Where it saves state

Everything lives in ./gsd-state.json in your project root. Git-committable, human-readable, always in sync.

Daily Workflow

The three commands you'll live in

/gsd:resume-work

Picks up exactly where you left off

Reads your state file, summarizes completed tasks, and primes Claude on what's next. Run this at the start of every session.

/gsd:execute-phase

Runs the current phase

Tells Claude to work through the active phase tasks one by one, checking each off as it completes. State updates automatically.

/gsd:progress

Snapshot of where you are

Prints completed phases, current tasks, blockers, and estimated remaining work. Use it anytime you feel lost.

Choosing the right mode

Quick vs. full GSD

Use /gsd:quick when...

✓

You have a single, well-defined task that takes under an hour

✓

You're adding a feature to an existing, stable project

✓

You don't need persistent state across sessions

Use full GSD when...

→

You're building something that spans multiple sessions or days

→

The project has multiple distinct deliverables or milestones

→

You want a shared plan that teammates can see and contribute to

Task Definition

What makes a task good

Bad task
task: "Work on the auth stuff" # Vague. No output defined. # Claude will guess what # "auth stuff" means and # probably guess wrong.

No clear scope, no acceptance criteria, no way to know when it's done. Claude interprets this as broadly as possible.

Good task
task: "Implement email/password login" accepts: | - POST /auth/login returns JWT - Invalid creds return 401 - Token stored in AsyncStorage - Redirects to /home on success

Specific deliverable. Acceptance criteria are testable. Claude knows exactly when it's done and can't scope-creep.

State Management

Pause anytime. Resume exactly where you stopped.

GSD saves state to gsd-state.json after every task completes. The file is plain JSON — readable, diffable, and safe to commit.

When you start a new session, /gsd:resume-work reads the file and reconstructs context for Claude without you having to re-explain anything.

// gsd-state.json { "project": "eMotus", "currentPhase": 2, "phases": [ { "id": 1, "name": "Foundation", "status": "complete" }, { "id": 2, "name": "Core Screens", "status": "in-progress", "tasks": [ { "name": "Home screen", "done": true }, { "name": "Lesson view", "done": false }, { "name": "Progress tracker", "done": false } ] } ] }
Know the limits

When GSD is overkill

🐛

Quick bug fix

A single broken thing. You know what's wrong. Just fix it.

→ Use /gsd:quick or just prompt directly
🔭

Exploratory spike

You don't know what you're building yet. Planning is premature.

→ Explore first, GSD when the shape is clear
📄

Single-file change

Editing one component, one config, one query. No phases needed.

→ Prompt directly with full file context
