The Claude Code Playbook

Claude Code — Master Prompts

5 prompts. Every session covered.

Prompt 1 / 5 When to use: Start of every session

Project Kickoff

I'm working on [project name] — [one sentence description]. Stack: [your stack] Key files: [file paths] Before we start, please: 1. Read [key file] to understand the current structure 2. Summarize what you understand about the project 3. Ask any clarifying questions before we build anything Today's goal: [what you want to accomplish this session]

Sets shared context before any code is written — prevents Claude from making assumptions about your project structure.

Prompt 2 / 5 When to use: Something broke

Debug Mode

I have a bug. Here's exactly what's happening: Expected: [what should happen] Actual: [what is happening instead] Error: [paste the full error message] Context: - This started happening after: [what changed] - Relevant file: [file path] Before suggesting a fix, please: 1. Read the file and understand its current state 2. Explain what you think is causing the bug 3. Ask if your theory matches my understanding 4. Then propose the fix

Forces diagnosis before prescription — you get a fix that actually addresses root cause, not just symptoms.

Prompt 3 / 5 When to use: Before every commit

Code Review

Please review [file or diff] before I commit this. I want you to check specifically for: 1. Bugs or logic errors 2. Edge cases I haven't handled 3. Anything that will break with unexpected input 4. Security issues (if relevant) What this code does: [one sentence] Please be direct. Don't soften feedback.

The last line matters — without it, Claude softens criticism. This unlocks honest, actionable review feedback.

Prompt 4 / 5 When to use: Cleaning up existing code

Refactor

I want to refactor [file or component name]. Goal: [what you want the refactored version to achieve] Constraints (do not change these): - The external API / props interface must stay the same - Do not change [specific thing] Before making any changes: 1. Read the current file 2. Describe what you plan to change and why 3. Call out any risks or side effects 4. Wait for my confirmation before writing code

Step 4 is the safety net — never let Claude refactor without showing its plan first. One confirmation saves hours of untangling.

Prompt 5 / 5 When to use: Building something new

Feature From Spec

I need to build: [feature name] What it should do: - [behavior 1] - [behavior 2] What it should NOT do: - [anti-requirement] Files that will likely be involved: [list the files] Before writing any code: 1. Read those files to understand existing patterns 2. Propose an implementation plan 3. Flag any questions or unclear requirements 4. Wait for approval before proceeding

Anti-requirements are as important as requirements — telling Claude what NOT to build prevents scope creep and unwanted "helpful" additions.
